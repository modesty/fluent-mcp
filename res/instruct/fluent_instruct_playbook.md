# Instructions for Fluent PlaybookDefinition API
Always reference the PlaybookDefinition API specifications for more details.
1. Import `PlaybookDefinition`, `wfa`, `PlaybookTriggerTypes`, and `ActivityDefinitions` from `@servicenow/sdk/automation`. Import Column types (`ReferenceColumn`, `IntegerColumn`, `StringColumn`, etc.) for inputs/outputs from `@servicenow/sdk/core`.
2. Use a Playbook to model a **guided, record-driven multi-step process** (lanes of activities a fulfiller works through). It is distinct from a Flow: playbooks are interactive and operate on a parent record.
3. Call `PlaybookDefinition(config, triggers, body)` with **three arguments**. `config` holds top-level properties plus declarative `inputs`/`outputs`; `triggers` is `{ triggers: [...] }`; `body` is `{ lanes: (params) => ({...}) }`.
4. In `config`, `$id` (use `Now.ID['...']`) and `label` are mandatory. Set `parentTable` to the table the playbook runs on — this auto-creates a `parent_record` input exposed as `params.parentRecord` (dot-walkable to fields). Treat `name` as stable identity: don't change it after deploy, or the platform creates a new record instead of updating.
5. The `triggers` array is **required even when empty** — pass `{ triggers: [] }` for a playbook with no triggers. Build each trigger with `wfa.playbook.trigger(triggerType, config, triggerInputs, playbookInputs?)`. Choose `triggerType` from `PlaybookTriggerTypes` (`RecordCreate`, `RecordUpdate`, `RecordCreateOrUpdate`, `Scheduled`).
6. Trigger inputs differ by type: record-based triggers take `{ table, condition?, runTriggerOnExtendedTables? }`; scheduled triggers take `{ table, condition?, limit (1-1000), startDateAndTime, timeZone? }` plus schedule-type fields. Use the optional 4th `playbookInputs` arg only when the playbook declares inputs or the trigger table differs from `parentTable`; wrap record references in `wfa.playbook.dataPill(trigger.current...)` and use plain values for literals.
7. `body.lanes` **must be a callback** `(params) => ({...})` — the build transformer reads lane/activity keys statically from the arrow-function body, not from a plain object. Build lanes with `wfa.playbook.lane({ config, activities })`, where `config` is a plain `LaneConfig` and `activities` is another `(params) => ({...})` callback. Inside `activities`, use `const` bindings and an explicit-key `return { name: name }` so keys are statically extractable.
8. Every lane and activity config requires `$id`, `label`, `order`, `startRule`, and `restartRule`. Set execution order with `startRule`: `wfa.playbook.run.Immediately()` for no dependencies, or `wfa.playbook.run.After(...deps)` (varargs, not an array) to wait for activities, lanes, or decision branches. `order` controls only visual layout. `restartRule` is `'RUN_ALWAYS'`, `'RUN_ONLY_ONCE'`, or `'RUN_ONLY_ON_RESTART'`.
9. Create activities with `wfa.playbook.activity(definition, config, inputs?, experienceProperties?)` using definitions from `ActivityDefinitions.Core.*` (e.g. `UpdateRecord`, `CreateNewRecord`, `SendEmail`, `Instruction`, `Decision`). Pass automation inputs in the 3rd arg and UI configuration in the 4th. Reference runtime data with `wfa.playbook.dataPill(...)` and field templates with `TemplateValue({...})`.
10. For branching, use `ActivityDefinitions.Core.Decision` with inputs `{ type: 'match_first' | 'match_all', branches: [...] }`. Each branch needs `id`, `label`, and (except the `else` branch) a `condition`. The `else` branch must be **last** with `id: 'else'`, `label: 'Else'`, and no condition. Depend on a branch via `wfa.playbook.run.After(decision.branches.<id>)`.
11. `outputs` are declarative schema only — you cannot assign or consume them from the Fluent DSL yet; wire them via the "Set Playbook Outputs" activity in Playbook Designer.
12. Remember: deploying a playbook does **not** update already-running instances. Open Playbook Designer and re-activate the playbook to apply changes at runtime (the build output logs this reminder).
